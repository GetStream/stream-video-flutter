# Making `stream_video` Jaspr-compatible — Production Rollout Plan

> **Execution step 0:** copy this file to the repository root as `JASPR_PRODUCTION_PLAN.md` so it can be shared (plan mode only permits writing the designated plan file; the repo-root copy is the first action after approval). Leave the earlier `JASPR_SUPPORT_PLAN.md` (the experiment) in place as a historical record — do not delete it — and link to it from the new doc's context section.

## Context

An experiment (`experimental/jaspr-support` branch, plan doc `JASPR_SUPPORT_PLAN.md`, 5 commits) already **proved this works end to end**: `stream_video` was made Dart-only, a `stream_video_jaspr` UI package and a `dogfooding_jaspr` demo app were built, and `jaspr build` now succeeds and produces a real compiled JS bundle. This plan turns that proven experiment into a real, mergeable, released change — same architecture, same code (largely reusable/rebasable from the experimental branch), but with production discipline: correct semver, CHANGELOG entries, a CI guard so this can't silently regress, durable fixes for a couple of hand-patched generated files, and a phased PR sequence instead of one long-lived branch.

**Why this needs its own plan and isn't just "merge the branch":** the experiment surfaced several things that only matter for a *real* release and weren't part of the original scope:
1. **`dart analyze` + `flutter test` are not sufficient verification.** They both still have access to `dart:io`/the full SDK, so they never caught two real Flutter/native leaks that only surfaced when actually running `jaspr build` (which compiles parts of the app in VM/native mode even for a client-web project): a `dart:ui`/`sky_engine` crash from `flutter: sdk: flutter` still being a *regular* (non-dev) dependency, and `dart_webrtc`/`package:web` being imported unconditionally. **The real acceptance test is a Jaspr build, and it needs to be a permanent CI job**, not a one-off manual check.
2. **Two of `stream_video`'s public API changes are breaking** (`NetworkMonitorSettings.internetConnectionInstance` removed; `AudioConfigurationPolicy`'s custom Android/Apple config now takes neutral types instead of plugin types) — confirmed via user decision: **ship as breaking, bump `stream_video` to `2.0.0`**, not deprecate-first.
3. **Two generated-code patches aren't durable.** `stream_video`'s OpenAPI client is generated by an external Docker image (`GetStream/openapi-generator`) with no local template — the `dart:io`-removal patches applied during the experiment will be **silently wiped by the next regeneration** unless we (a) extend the repo's existing `MANUAL_EDIT` + regression-test convention to cover them, and (b) file the underlying fix upstream.
4. **Downstream version-constraint fallout**: 6 packages/apps in this monorepo pin `stream_video: ^1.4.1` / `stream_video_flutter: ^1.4.1` and the root `melos:` bootstrap-sync block does too — all need bumping in lockstep with the major version.
5. **A CI/tooling footgun**: `jaspr build` silently *skips* the JS compile (warning only, not an error) if `build_web_compilers` isn't an explicit dev dependency — the CI guard must assert the output file actually exists, not just that the command exit code was 0.

**Decisions already made (do not re-litigate):**
- **Breaking change, major version bump**: `stream_video` → `2.0.0`. `stream_video_flutter` bumps in lockstep to `2.0.0` too (its own public API is unchanged, but its `stream_video: ^2.0.0` constraint changes and lockstep major versions keep the SDK family's versioning legible — this repo already releases both packages together per `chore(repo): Release vX.Y.Z` PRs).
- **Phased PRs**, mirroring the phases below, reviewed and merged in sequence — not one giant PR.
- **`stream_video_jaspr` stays internal** (workspace/path package, `publish_to: none`) for now — not published to pub.dev this round.
- **Ship the first-cut UI scope validated in the experiment** (camera grid, mic/camera toggle, leave) — screenshare/PiP/reactions/SSR are explicit, non-blocking follow-ups.

**Reuse the experiment.** The `experimental/jaspr-support` branch's 5 commits are already `melos analyze`/`test` clean and proved out with a real `jaspr build`. Rather than re-deriving the diffs from scratch, each PR below should **cherry-pick/rebase the corresponding commit(s)** from that branch and layer the production-only additions (CHANGELOG, version bumps, CI guard, durable codegen fix, downstream constraint bumps) on top. Delete the experimental branch once all PRs have landed.

## Architecture (proven, unchanged from the experiment)

`packages/stream_video/lib/src/platform/` holds **abstract interfaces + a registry whose defaults are pure-Dart/web**, mirroring the existing `PlatformInterface` idiom already used by `stream_video_flutter`:

```dart
abstract class StreamWebRtc { ... }            // engine globals + native factory + helper + initialize
abstract class NetworkMonitor { ... }          // internet/connectivity checking
abstract class NetworkMonitorFactory { ... }   // builds the NetworkMonitor for a StreamVideo instance
abstract class DeviceStateProvider { ... }     // battery + thermal
abstract class EnvironmentInfoProvider { ... } // device/package/system info
abstract class AppLifecycleProvider { ... }    // app lifecycle (WidgetsBinding)
abstract class DisplayMetricsProvider { ... }  // physical screen size
```

Each exposes `static X instance = <PureDartDefault>();`. `stream_video_flutter`'s `StreamVideoFlutter.ensureInitialized()` (already wired into the existing native bootstrap path, so app authors need zero code changes) registers all native/plugin-backed implementations in one call; web/Jaspr registers nothing and gets working defaults. The WebRTC default throws on native platforms (fail loud, not silent no-op) if `ensureInitialized()` was never called.

## PR 1 — Cosmetic quick-wins + provider registry scaffold + `webrtc_interface` swap

*Rebase from experimental commits `88517ca6` (Phase 0), `023d4077` (Phase 1), `8bd327df` (Phase 2). Purely additive/behavior-neutral — safest PR to land first.*

- Cosmetic Flutter-reference removal (`debugPrint`→logger, `@immutable`→`package:meta`, `VoidCallback`→`void Function()`, vestigial `flutter/foundation` import).
- Add `platform/` interfaces + pure-Dart defaults + `StreamVideoFlutter.ensureInitialized()`. Nothing consumes them yet.
- Swap the 16 qualifying `src/webrtc/**` files from `stream_webrtc_flutter` to `webrtc_interface` directly (pure abstract Dart, zero platform code — safe for both native and web targets). **Do not** attempt a direct `dart_webrtc` swap for files needing concrete factory globals — the experiment found `dart_webrtc`'s implementation is unconditionally JS-interop-only (no native/VM branch) and broke `flutter test` immediately when tried; `webrtc_interface` alone covers these 16 files.
- Keep `stream_webrtc_flutter` in `stream_video`'s pubspec for now.

**Verification:** `melos run analyze:error` + `melos run test:all` + `melos run format:verify` green across the whole workspace (not just `stream_video`).

## PR 2 — WebRTC engine injection, drop `stream_webrtc_flutter` from `stream_video`

*Rebase from experimental commit `12322b05` (Phase 3).*

Introduce `StreamWebRtc` for the 9 remaining files needing concrete factory globals/plugin-only symbols (`call.dart`, `audio_configuration_policy.dart`, `peer_connection_factory.dart`, `rtc_manager.dart`, `rtc_media_device_notifier.dart`, `rtc_local_track.dart`, `media_constraints.dart`, `stream_video.dart`, `media_frame_reporter.dart`):

```dart
abstract class StreamWebRtc {
  Future<RTCPeerConnection> createPeerConnection(Map cfg, Map constraints);
  StreamWebRtcHelper get helper;
  StreamNativeFactory? createNativeFactory(...);  // null on web
  Future<void> initializeAudioConfiguration({required bool refresh, required Map options});
}
```

- Web/Jaspr default `DartWebRtcEngine` (in `stream_video`) — reached **only** via `if (dart.library.js_interop)` conditional file selection (`stream_web_rtc_default.dart` throwing native stub vs `stream_web_rtc_default_web.dart`), never an unconditional `dart_webrtc` import.
- Native `FlutterWebRtcEngine` (in `stream_video_flutter`), registered in `ensureInitialized()`.
- `NativePeerConnectionFactory`→`StreamNativeFactory?`, `Helper`→`StreamWebRtcHelper`, `WebRTC.initialize`→`StreamWebRtc.initializeAudioConfiguration` (cross the boundary with `Map` payloads, never plugin enums), device enumeration routed through the engine.
- `AndroidAudioConfiguration`/`AppleAudioConfiguration` (breaking): new neutral `StreamAndroidAudioConfiguration`/`StreamAppleAudioConfiguration` + enum mirrors in `models/audio_configuration.dart`; `AudioConfigurationPolicy`'s public API returns these instead of plugin types.
- Remove `stream_webrtc_flutter` from `stream_video`'s main `dependencies:` (kept as a **dev_dependency** — the test suite needs a native-backed engine, and dev deps don't leak into consumers' resolution regardless).

**Verification:** same as PR 1, plus a **real Flutter web build** (`flutter build web` from `dogfooding` or `stream_video_flutter/example`) to prove `dart_webrtc` actually compiles under dart2js, not just `dart analyze`.

## PR 3 — Telemetry/network decoupling, drop the Flutter SDK dependency, durable codegen fixes, CI guard

*Rebase from experimental commit `d723497b` (Phase 4+5), which already includes the fixes below — this PR just needs the production hardening layered on top.* **This is the PR that actually unblocks Jaspr**, and the riskiest one — treat every sub-item as required, not optional.

**Telemetry (`stream_video` → `stream_video_flutter`):**
- `EnvironmentInfoProvider` (device/package/system info), `DeviceStateProvider` (battery/thermal, neutral `StreamThermalStatus` enum) — both already scaffolded in PR 1, just need their native impls' consumption sites migrated and `battery_plus`/`thermal`/`device_info_plus`/`package_info_plus`/`system_info2`/`connectivity_plus`/`internet_connection_checker_plus` moved out of `stream_video`'s pubspec into `stream_video_flutter`'s (which needs the two network ones added — it didn't already have them).
- **`NetworkMonitor` promotion (breaking, highest risk)**: add `currentStatus`/`checkInterval`/`setIntervalAndResetTimer` to the interface; replace `InternetConnection`/`InternetStatus` with `NetworkMonitor`/`NetworkStatus` across `call.dart` (incl. `_awaitNetworkAvailable` reconnection logic), `sfu_ws.dart`, `call_session*.dart`, `coordinator_ws.dart`, `coordinator_client_open_api.dart`. Web default: `HttpNetworkMonitor` (periodic `http` HEAD polling). Native: `FlutterNetworkMonitorFactory` wraps `InternetConnection`+`Connectivity`, moved into `stream_video_flutter`.
- **Breaking**: `NetworkMonitorSettings.internetConnectionInstance` (plugin type) removed outright, replaced with a `networkMonitor` field of the new neutral type.
- **Test-suite gotcha to pre-empt**: mocktail's `any()` on `NetworkMonitor.setIntervalAndResetTimer(Duration)` needs `registerFallbackValue(Duration.zero)` registered — missing this caused ~35 test failures in the experiment before being diagnosed. Add it to `call_test_helpers.dart`'s `registerMockFallbackValues()` up front this time.

**Lifecycle/display + drop the Flutter dependency:**
- `AppLifecycleProvider`/`DisplayMetricsProvider` (already scaffolded in PR 1) — migrate `rtc_manager.dart`'s `WidgetsBinding` usage and delete `lifecycle/lifecycle_utils_io.dart`, replacing it with a single `lifecycle_utils.dart` that delegates to `AppLifecycleProvider.instance`.
- **Remove `flutter: sdk: flutter` from `stream_video/pubspec.yaml`'s `dependencies:`** (this is the actual fix — keep `environment: { flutter: ... }` and `flutter_test`/`stream_webrtc_flutter` as **dev_dependencies only**, since dev deps don't leak into any consumer's pub workspace resolution). This single change is what the experiment found actually matters — Dart Pub Workspaces unify resolution across all members, so a regular (non-dev) `flutter:` dependency anywhere pulls `dart:ui`/`sky_engine` into *every* workspace member's graph, including a pure-Dart Jaspr app that never imports Flutter.

**Durable fixes for two unconditional `dart:io`/JS-interop leaks found only by actually running `jaspr build`** (`dart analyze`/`flutter test` do not catch these):
- `stream_video`'s public barrel unconditionally exported `src/logger/impl/file_logger.dart` (genuinely native-only, uses `dart:io`'s `File`/`Directory`) — move it wholesale to `stream_video_flutter/lib/src/platform/file_logger.dart`, re-export from `stream_video_flutter`'s barrel instead.
- The generated OpenAPI client (`lib/open_api/video/coordinator/{api,api_client,api/product_video_api}.dart`) uses `dart:io`'s `HttpStatus`/`SocketException`/`TlsException`/`IOException` unconditionally. **These files are regenerated from an external Docker image (`ghcr.io/getstream/openapi-generator`) with no local template in this repo** — a raw patch will be silently wiped on the next `generate-openapi-stream.sh` run. To make it durable:
  1. Replace `HttpStatus.badRequest`/`.internalServerError`/`.noContent` with local `const int` constants (`api.dart`, placed after the `part` directives — they must come after, not before), and drop the redundant `SocketException`/`TlsException`/`IOException` catch clauses in `api_client.dart` (they're all `Exception` subtypes that already fall through correctly to the generic `on Exception catch` below — only the error message text changes). Mark every edit `// MANUAL_EDIT`, matching this repo's existing convention for hand-patched generated files.
  2. **Extend `test/src/open_api/open_api_manual_edits_test.dart`** (the repo's existing "verify MANUAL_EDIT patches survive regeneration" test file) with cases asserting `ApiException` still carries the right status codes (400/500/204) and that no `dart:io` symbols are referenced — so a future regeneration that wipes these patches fails CI immediately instead of silently reintroducing a Jaspr-breaking leak.
  3. **File an issue against `GetStream/openapi-generator`** asking the Dart generator template to stop emitting `dart:io` (or gate it behind a template flag) — this is the actual fix; steps 1-2 are the interim safety net.
  4. Add a one-line comment atop `generate-openapi-stream.sh`/`generate-openapi-origin.sh` pointing at the manual-edits test file as a post-regen checklist.

**New CI guard (this is what makes the decoupling durable, not just true-today):**
- Add a `jaspr_build_guard` job (new lightweight workflow file, e.g. `.github/workflows/jaspr_build.yml`, triggered on `pull_request` + `push: main` matching this repo's convention): install the Dart SDK, `dart pub get` at the workspace root, `dart pub global activate jaspr_cli`, `cd dogfooding_jaspr && jaspr build`, then **assert `build/jaspr/main.client.dart.js` exists and is non-trivially sized** (the experiment found `jaspr build` exits 0 and only *warns* if the JS compile step is silently skipped — e.g. missing `build_web_compilers` dev dependency — so checking the exit code alone is not sufficient).
- Confirm `dogfooding_jaspr/pubspec.yaml` has `build_web_compilers` as an explicit dev dependency (easy to forget; `jaspr_cli` doesn't add it automatically to a hand-authored pubspec).

**Verification:** full `melos run analyze:error`/`test:all`/`format:verify`, plus the new `jaspr_build_guard` passing, plus a Flutter **web + desktop** manual smoke test (network-monitor/reconnection changes are the highest-risk item here).

## PR 4 — `stream_video_jaspr` package (internal)

*Rebase from experimental commit `edaaf8b9` (Phase 6), which already includes the js-interop-gating fix below.*

```
packages/stream_video_jaspr/
  pubspec.yaml (publish_to: none — internal for now)
  lib/stream_video_jaspr.dart                      # barrel
  lib/src/call/call_state_builder.dart              # StateEmitter<CallState> -> setState bridge
  lib/src/renderer/{video_element,video_element_default,video_element_default_web,video_track_renderer}.dart
  lib/src/renderer/stream_video_view.dart           # track selection + minimal dynascale
  lib/src/participants/{participant_grid,participant_tile}.dart
  lib/src/controls/call_controls.dart
  lib/src/call/stream_call_content.dart
```

**Key lesson baked in from the experiment**: `jaspr_builder`'s style-extraction pass compiles in **VM/native mode even for a client-web project**. Any unconditional `dart_webrtc` or raw `package:web` import — even in code that only ever *runs* in a browser — breaks that pass. `video_track_renderer.dart`'s video element must go through a `StreamVideoElement` abstraction gated behind `if (dart.library.js_interop)` (mirroring `stream_video`'s own `stream_web_rtc.dart` pattern exactly), and any `GlobalNodeKey`/DOM type must come from `package:universal_web` (conditionally web-safe), never `package:web` directly.

**Scope (per user decision — do not expand this round):** local + remote camera tiles in a CSS grid, mic/camera toggle, leave. Screenshare, spotlight/PiP, reactions, and SSR are explicit follow-ups, tracked as a short "Not in this round" list in the package README.

**Verification:** `dart analyze` clean; builds successfully as part of `dogfooding_jaspr`'s `jaspr build` (see PR 3's CI guard).

## Interop contract (unchanged from the experiment, restated for reference)

A Flutter and a Jaspr client can join the same call when both share: the same **apiKey**, the same **environment host**, the same **call type**, the same **call id**, and a compatible **token** (same backend, same `/api/auth/create-token`). Both talk to the same coordinator + SFU through the same `stream_video` core; only the renderer differs.
